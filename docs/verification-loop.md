# 검증 루프 설계 템플릿

> CLAUDE.md §3에 넣을 검증 루프를 처음 설계할 때 사용. 채운 뒤 이 파일은 상세 규약 보관소가 된다.

## 1. 게이트 구성 (순서 고정)

| 순서 | 게이트 | 명령 | 잡는 것 |
|---|---|---|---|
| 1 | 타입체크 | {{예: npx tsc --noEmit}} | 형식 오류 — 가장 싸고 빠름 |
| 2 | 전체 테스트 | {{예: npx vitest run}} | 불변식·회귀 |
| 3 | 빌드 | {{예: npm run build}} | 번들·환경 문제 |
| 4 | 프로젝트 특화 게이트 | {{예: id 중복 검사, 커버리지 프로브, 시크릿 스캔}} | 도메인 무결성 |
| 5 | 육안 확인 | (명령 아님) | 기계가 판정 못 하는 품질 |

## 2. 기준선 관리

- **테스트 수 기준선**을 CLAUDE.md §3에 날짜와 함께 기록. **줄어들면 뭔가 깨진 것.**
- 테스트를 추가한 작업(Phase)이 기준선을 갱신할 의무를 진다.
- 기준선의 정본 위치를 한 곳으로 고정 (마스터플랜 상태판 또는 CLAUDE.md §3 — 두 곳에 두면 어긋난다).

## 3. 커버리지 프로브 패턴 (사각지대 방지)

개별 항목 테스트는 "등록된 항목"만 본다 — **미등록 항목은 미검사로 통과**하는 사각지대가 생긴다.
전역 프로브를 하나 만들어라:

```
"모든 {{항목}}이 {{레지스트리}}에 등록되어 있다" (전수 순회 대조)
"등록된 모든 항목이 {{필수 메타/테스트}}를 갖는다"
```

예 (DRACONIS 실전): 신규 메타 파일을 인덱스에 배선 안 하면 그 스킬이 메타 검사에서 조용히 빠짐
→ "전 스킬이 메타에 태깅됨" 전역 프로브가 적발.

## 4. 렌더/스모크 테스트 패턴

산출물이 UI로 렌더된다면: **전 항목을 실제 렌더 함수로 통과**시키는 스모크 테스트 하나가
"앱을 안 켜고도" 렌더 크래시·미지원 토큰을 잡는다. (예: renderToString 무-throw 전수 검사)

## 5. 부분 실행의 함정

- 부분 실행(`-t` 필터)은 개발 중 반복용일 뿐, **통과 근거가 아니다.**
- 전체 실행이 필수인 검사(교차 파일 감사·전역 프로브)를 명시적으로 열거하고 "필터 금지"를 규약에 박는다.

## 6. 임시 검증 파일 규약

- 임시 테스트·덤프 파일 접두사를 정한다 ({{예: `zztmp_*`}}) — 자동정리 훅과 충돌하지 않는 이름으로.
- 제출(커밋) 전 전부 삭제. verify 스킬이 `git status`로 확인.
