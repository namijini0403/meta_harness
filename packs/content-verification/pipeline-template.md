# 대량 저작 파이프라인 설계 템플릿

> "같은 종류의 콘텐츠를 수십~수백 개 추가"가 반복되는 프로젝트의 표준 절차.
> 실전 검증: DRACONIS 학년 콘텐츠 파이프라인(12학기·스테이지 312개·테스트 35,000+를 이 구조로 생산).

## 절차 (5단계)

### 1. 청사진 작성 (메인)
- 산출 단위 목록을 문서로 먼저 고정: `docs/{{청사진}}.md` — id·이름·수량·의존 관계.
- 기존 청사진과 **같은 형식**으로 (형식 통일 = 다음 에이전트가 참조 구현으로 사용).

### 2. 충돌 사전 확인 (메인)
- 새 id/접두사/키가 기존 레지스트리와 충돌하는지 grep으로 전수 확인. 충돌 시 청사진 수정.
- 이 단계를 건너뛰면 에이전트 산출물을 통째로 재작업하게 된다.

### 3. 구현 에이전트 가동 (메인이 브리프 발송)
- 배치 크기: 에이전트당 {{예: 단원 3개}} — 너무 크면 품질 저하·죽으면 손실 큼, 너무 작으면 오버헤드.
- **공유 레지스트리 파일을 만지는 에이전트는 순차** (첫 에이전트가 항목 생성, 다음이 append).
- 브리프 = `docs/agent-brief-template.md` 8요소 + 완성 세트 기준 + 자체 검증 루프.
- 에이전트가 죽으면: `git status`로 산출물 확인 → 이어받기 에이전트("기존 산출물 검수부터").

### 4. 통합 검증 (메인)
- CLAUDE.md §3 전체 게이트 + 육안 확인.
- 병렬 에이전트 간 교차 중복(같은 항목을 둘이 만듦)은 불가피 — **중복 검사 게이트를 두고 메인이 한쪽 제거.**

### 5. 커밋 + 후속 명세 (메인)
- 커밋 규약(§7). 외부 협업 산출물(이미지 등)이 필요하면 명세 문서만 작성(§4) — 폴백이 있어 블로커 아님.

## 완성 세트 정의 (프로젝트마다 채우기)

| # | 산출물 | 규약 |
|---|---|---|
| ① | {{본체 — 예: 생성기/문서/모듈}} | {{불변식}} |
| ② | {{설명/메타 — 예: 해설, 태그}} | {{형식}} |
| ③ | {{등록 — 예: 레지스트리 append}} | {{순차 규칙}} |
| ④ | {{검증 — 독립 재계산 테스트}} | 제출 후 삭제 or CI 편입 |

하나라도 빠지면 커밋 불가. 가능하면 "미완 세트 = 테스트 실패"로 강제 (커버리지 프로브).

## 통제 어휘 패턴

여러 산출물이 공유하는 분류 토큰(오류 코드·태그·카테고리)은 **한 파일에서 통제**한다:
- 산출물은 통제 어휘에 **실재하는 토큰만** 사용 (자유 문자열 금지 — 테스트로 강제).
- 새 토큰이 필요하면 에이전트는 **PROPOSED로 보고만** → 메인 승인 후 등록. (무단 확장 = 동기화 지점 붕괴)
