# packs — 선택 모듈 카탈로그

> 코어(CLAUDE.md + .claude/ + docs/)는 필수. 팩은 **필요한 것만 골라 프로젝트에 넣고, 안 쓰면 폴더째 삭제**한다.
> 나중에 필요해지면 meta_harness repo에서 해당 팩만 다시 복사하면 된다 — 팩은 서로 **하드 의존 없음**(단독 설치·제거 가능. db-analytics ↔ security-privacy처럼 "병용 권장" 소프트 참조만 있음).

## 설치 / 제거

- **설치**: 팩 폴더를 프로젝트에 복사 → 팩 안의 `INSTALL.md` 절차대로 (① 스킬/문서를 제 위치로 이동 ② CLAUDE.md에 해당 절 추가 ③ 플레이스홀더 채우기).
- **제거**: 팩이 설치한 스킬·문서 삭제 + CLAUDE.md의 해당 절 삭제. 팩은 코어를 수정하지 않으므로(추가만) 제거가 안전하다.

## 카탈로그

| 팩 | 언제 넣나 | 내용물 |
|---|---|---|
| **content-verification** | 콘텐츠(문항·문서·데이터·코드 산출물)를 대량 생산하는 프로젝트 | 저작 파이프라인 스킬 템플릿(완성 세트 강제) · 감사 스킬 템플릿(읽기전용 스캐너) · 3중 검증(자동+독립재계산+육안) 절차 · 파이프라인 설계 템플릿 |
| **db-analytics** | DB에 사용자 데이터를 저장하거나 학습분석·이벤트 수집을 하는 프로젝트 | 레지스트리 패턴 DDL 템플릿(app/event_type 버전관리) · 운영/분석 DB 분리 원칙 · 이벤트 화이트리스트 3곳 동기화 규약 · 자유텍스트 마스킹·no-op 법적 안전장치 |
| **security-privacy** | 개인정보·아동·학교·의료 등 민감 도메인, 또는 외부 공개 서비스 | 보안 문서 스위트 목차(STRIDE·ASVS L2·RBAC/ABAC·데이터분류·보존/파기·인시던트 러닝북) · 보안 감사 에이전트 정의 · RLS 정책 템플릿 · 동의 게이트 설계 · Anthropic Cybersecurity Skills 연동 가이드 |
| **code-collab** | 같은 파일의 병렬 실험(A안/B안 경쟁)이나 코드리뷰 흐름이 필요한 프로젝트 | git worktree 병렬 절차(격리·수확·정리) · 코드리뷰 프로토콜(요청/수행/수신 — 리뷰어 독립성·방어 금지 규율) |
| **e2e-deploy** | 브라우저 앱(웹/PWA)이 있거나 자동 배포를 쓰는 프로젝트 | Playwright 설치→작동 절차 · 스모크 스위트 골격(콘솔 에러 0 포함) · 브라우저 검증 2모드 규약(자동 E2E vs AI 조작 육안) · **실행 가능한 배포 폴링 스크립트**(버전 마커로 구버전 오검증 방지) · 배포 검증 루프·롤백 실험·CI 잡 |
| **design-skills** | 웹 UI·랜딩 페이지·모션이 있는 프로젝트 | MengTo/Skills(MIT) 큐레이션 — 보안·라이선스 감사 기록(기준 커밋 고정) · 세트 A 디자인 코어(자동 발동 7종) · 세트 B 레퍼런스→프롬프트 워크플로(5종) · 세트 C 스타일 카탈로그 62종 + **라우터 스킬**(스킬 목록 비대화 없이 자동 배차) |

## 팩 조합 가이드

- 학습앱·연구 플랫폼 (DRACONIS류): content-verification + db-analytics + security-privacy + e2e-deploy.
- 사내 도구·CLI: content-verification만 (산출물 검증 규율).
- 공개 SaaS: db-analytics + security-privacy + e2e-deploy (+팀 개발이면 code-collab).
- 정적 사이트·문서 프로젝트: 팩 없이 코어만.
- code-collab은 어느 조합에든 추가 가능 — 기본 병렬(소유 경계)로 부족할 때만.
- e2e-deploy는 브라우저 UI 또는 자동 배포 중 하나라도 있으면 권장 — 둘 다 없으면(순수 라이브러리) 불필요.
- design-skills는 사용자 대면 웹 UI가 있으면 세트 A만이라도 — 랜딩/마케팅 페이지를 만들면 세트 C까지.
- **서드파티 스킬 편입 규약**: design-skills의 감사 기록 형식(기준 커밋 고정·스크립트 전수·인젝션 스캔·도메인·라이선스)이 선례 — 다른 외부 스킬 모음을 들일 때도 같은 감사 후 팩으로.

## 새 팩 만들기

패턴이 한 프로젝트에서 검증되면 팩으로 승격한다. 팩의 구성 규약:

```
packs/<pack-name>/
├─ INSTALL.md      ← 설치 절차(어디에 무엇을 복사, CLAUDE.md에 무슨 절 추가) + 제거 절차
├─ <문서 템플릿들>.md
└─ skills/         ← (있으면) .claude/skills/로 복사할 스킬
```

규칙: 팩은 **코어를 수정하지 않는다**(CLAUDE.md에 절 "추가"만 — INSTALL.md에 추가할 절 전문을 수록). 팩끼리 의존하지 않는다.
