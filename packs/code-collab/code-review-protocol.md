# 코드리뷰 프로토콜 (요청 / 수행 / 수신)

> 자기 코드는 자기가 못 본다 — 구현자와 리뷰어의 **분리**가 리뷰 가치의 전부다.
> 리뷰어 = 별도 에이전트(read-only) 또는 별도 세션. 구현 컨텍스트가 없는 눈이어야 한다.

## 1. 요청 (구현자 → 리뷰어)

리뷰 요청에 담을 것:
- **diff 범위** (브랜치·커밋 구간) + 변경 요약 (무엇을 왜)
- **스펙/계획서 링크** (plan-template §0 성공 기준 — 리뷰어는 "스펙대로인가"를 본다)
- **자기 신고**: 자신 없는 부분·타협한 부분을 먼저 밝힌다 (리뷰어 시간을 거기 집중)
- 리뷰어에게 **금지**: 코드 수정 (리포트만 — audit 패턴과 동일)

## 2. 수행 (리뷰어)

우선순위 순서로 본다 (아래로 갈수록 사소 — 위가 비었을 때만 아래를 지적):
1. **정합성**: 스펙과 다르게 동작하는가, 깨지는 입력이 있는가 (구체 실패 시나리오 필수 — "느낌상 위험"은 finding이 아니다)
2. **보안·데이터**: 인젝션·권한·시크릿·개인정보 (민감 도메인이면 security-auditor에 별도 위임)
3. **회귀 위험**: 기존 동작·저장 호환·불변 결정(decisions-log) 위반
4. **설계**: 소유 경계 침범·공유 파일 규약 위반·중복 구현 (기존에 있는 걸 또 만듦)
5. 스타일·명명 (직접 수정 제안 형태로만, 짧게)

리포트 형식 = 심각도(Critical/Important/Minor) · file:line · **실패 시나리오** · 수정안. 확신 없는 지적은 "확인 요망"으로 구분 표기.

## 3. 수신 (구현자 — 여기가 규율의 핵심)

- **방어 금지**: 지적마다 "그건 의도입니다"로 답하고 싶은 충동 = 리뷰 무효화. 각 finding을 ①수용(수정) ②반박(**재현 불가·스펙 근거** 제시) ③보류(사용자 판단) 셋 중 하나로 처리하고 목록으로 남긴다.
- **Critical/Important 미해소 상태로 병합 금지.** Minor는 묶어서 후속 커밋 가능.
- 리뷰어가 맞았던 지적 유형은 합리화 방지표에 기록 — 다음엔 구현 단계에서 잡는다.
- 수정 후 재검증: 수정 diff에 대해 §3 게이트 재실행 (리뷰 반영이 새 버그를 만드는 게 흔하다).
