# git worktree 병렬 작업

> 기본 병렬 전략은 소유 경계(파일이 안 겹치게)다. worktree는 **경계로 못 푸는 경우**의 도구:
> ① 같은 파일을 서로 다른 접근으로 병렬 실험 (A안/B안 경쟁 구현) ② 메인 작업 중 긴급 수정 끼어들기 ③ 대규모 마이그레이션을 앱 동작 유지본과 분리.

## 원리

worktree = 같은 저장소의 **독립 체크아웃 디렉터리** (브랜치별 물리 폴더). 에이전트마다 자기 worktree를 주면 같은 파일을 만져도 충돌하지 않는다 — 충돌은 병합 시점으로 미뤄지고, 병합 판단은 오케스트레이터가 한다.

## 절차

```bash
# 1. 생성 (프로젝트 밖 경로에 — 프로젝트 안에 만들면 빌드·검색이 이중으로 잡힌다)
git worktree add ../{{프로젝트}}-wt-{{작업명}} -b wt/{{작업명}}

# 2. 에이전트 브리프에 명시: "작업 디렉터리 = ../{{...}}-wt-{{작업명}}, 그 밖 접근 금지"
#    (worktree 안에서는 파일 경계 불필요 — 전체가 그 에이전트 소유)

# 3. 수확: 오케스트레이터가 각 worktree 결과를 검증(§3) 후 승자/병합 결정
git merge wt/{{작업명}}          # 또는 cherry-pick·수동 이식

# 4. 정리 (필수 — 잊으면 유령 브랜치·디스크 누수)
git worktree remove ../{{프로젝트}}-wt-{{작업명}}
git branch -d wt/{{작업명}}
git worktree prune
```

## 규율

- **worktree 수확·병합은 오케스트레이터 전담** — 에이전트가 자기 worktree에서 main으로 병합 금지 (커밋 금지 규약의 연장).
- 경쟁 구현(A안/B안)은 **판정 기준을 생성 전에** 정한다 (테스트 수·성능·코드량) — 결과 보고 나서 기준을 만들면 편향된다.
- node_modules 등 무거운 의존성은 worktree마다 재설치가 필요할 수 있다 — 실험이 가벼울 땐 경계 병렬이 여전히 싸다.
- 세션 종료 전 `git worktree list`로 잔존 확인 — 남은 worktree는 다음 세션의 "이게 뭐지" 비용.
