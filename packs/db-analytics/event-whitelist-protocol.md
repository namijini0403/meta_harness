# 이벤트 화이트리스트 — 3곳 동기화 프로토콜

> 이벤트는 **화이트리스트에 있는 타입·필드만** 수집한다. 목적: ① 무분별 수집 방지(법·윤리) ② 스키마 표류 방지 ③ "이 이벤트가 뭐였지"의 정본 확보.

## 동기화 3곳 (프로젝트 경로 기입)

| # | 위치 | 파일 | 역할 |
|---|---|---|---|
| 1 | 클라이언트 타입 | {{예: client/src/analytics/types.ts}} | 발신 가능한 타입·payload 형태 (컴파일 타임 강제) |
| 2 | 서버 허용 목록 | {{예: server/src/index.ts의 allowlist}} | 수신 게이트 — 목록 밖 타입은 거부 (런타임 강제) |
| 3 | DDL 레지스트리 | {{예: server/sql/analytics-schema.sql의 event_type_registry}} | 분석 측 정본 + 버전·스키마 기록 |

## 새 이벤트 추가 절차 (전부 한 커밋에)

1. **질문 먼저**: 이 이벤트로 답하려는 분석 질문을 한 줄로 적는다 (커밋 메시지/레지스트리 description에).
2. 3곳을 **한 커밋에서 동시 갱신** — 하나라도 빠지면 발신 실패 또는 조용한 드롭.
3. payload 검사: 자유 텍스트·닉네임·식별정보 0 (db-design-principles.md §4).
4. 기존 이벤트의 payload를 바꾸면 레지스트리 version +1 (기존 행 수정 금지).

## 소유 규칙 (병렬 안전)

- 이벤트 추가·수정은 **분석 담당 에이전트 단독** — 다른 에이전트는 이벤트를 추가하지 않는다 (필요하면 요청만).
- 이유: 3곳 동기화는 교차 파일 작업이라 병렬 충돌·부분 갱신이 잘 난다.

## 예외 규정

- 서버가 직접 발신하는 이벤트(사용자 클라를 거치지 않는 것)는 클라 타입(1번)에 없어도 된다 — 단 2·3번엔 반드시 등록하고, 그 사실을 레지스트리 description에 명시.
