# DB·분석 설계 원칙

## 1. 운영 DB / 분석 DB 물리 분리

- **운영 DB(#1)**: 서비스 동작에 필요한 상태 (계정·진행·설정). **분석 DB(#2)**: 연구·통계용 이벤트.
- 별도 프로젝트/인스턴스 + **접속 키 완전 분리**. 코드에서 커넥션 풀 혼용 금지 (풀 변수부터 분리: `pool` vs `analyticsPool`).
- 이유: ① 유출 시 피해 반경 최소화 ② 분석 부하가 운영을 못 건드림 ③ 법적 근거(동의)가 다른 데이터를 물리적으로 다르게 취급.

## 2. no-op 법적 안전장치

- 분석 DB URL 미설정 = **수집 코드 전체가 조용히 no-op**. 동의(학부모 동의·기관 승인 등) 완료 전에는 URL을 설정하지 않는 것만으로 수집이 원천 차단된다.
- **이 no-op을 우회하는 코드 금지** — 편의를 위한 로컬 폴백·임시 저장도 금지 (그게 곧 무동의 수집이다).
- 개통 순서 불변식: **수집 스키마가 완성되기 전 URL 설정 금지** ("첫날부터 완전한 데이터" 원칙 — 반쪽 스키마로 모은 데이터는 연구에 못 쓴다).

## 3. 레지스트리 패턴 (버전 관리)

- 열거형이 될 값(앱 id·이벤트 타입·오류 코드)은 코드 상수가 아니라 **DB 레지스트리 테이블**로: `(값, version, schema_json, deprecated)`.
- 스키마가 바뀌면 version +1 (기존 행 수정 금지 — 과거 데이터의 해석 근거 보존).
- `deprecated` 플래그로 신규 수집 차단 (삭제 금지).
- → `analytics-schema-template.sql`이 골격.

## 4. 자유 텍스트·식별정보 격리

- **이벤트에 자유 텍스트·닉네임·식별정보를 절대 넣지 않는다** (화이트리스트로 강제 — event-whitelist-protocol.md).
- 자유 텍스트(일기·메모 등)가 기능상 필요하면: **원문 = 운영 DB만**, 분석 DB엔 **마스킹본 + 파생 지표만**.
- 사용자 식별은 가명 id (운영 id와 분석 id의 매핑 테이블은 운영 측에만, 접근 최소화).

## 5. 이벤트 설계 체크리스트

- [ ] 이 이벤트로 답하려는 질문이 명확한가? (질문 없는 수집 = 부채)
- [ ] 화이트리스트 3곳 동기화 완료? (클라 타입 · 서버 허용 목록 · DDL 레지스트리)
- [ ] payload에 자유 텍스트·식별정보 0?
- [ ] 재현·순서 복원에 필요한 필드(seed·seq_no·정규화 응답 등)를 처음부터 넣었는가? (나중에 추가한 필드는 과거 데이터에 없다)
- [ ] 미동의 환경에서 no-op 확인?

## 6. RLS·보안 연계

행 수준 보안(RLS)·감사 로그·보존/파기 정책은 `packs/security-privacy/` 소관. 이 팩과 함께 쓸 때:
분석 DB에도 RLS 기본 거부 + 서비스 롤만 쓰기, 대시보드는 읽기 전용 뷰로.
