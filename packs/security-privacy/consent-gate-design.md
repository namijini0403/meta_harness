# 동의 게이트 설계 — 동의 전 수집 원천 차단

> 개인정보(특히 아동·연구 데이터) 수집은 동의가 **코드 구조로** 보장돼야 한다. "동의 받고 켜자"는 다짐이 아니라 아키텍처로.

## 핵심 패턴: 설정 부재 = no-op

```
수집 대상 DB의 접속 URL(환경변수)이 미설정이면
→ 수집 코드 전체가 조용히 no-op (에러도 아니고 큐잉도 아니고 그냥 아무것도 안 함)
```

- 동의·기관 승인 완료 **후에만** 운영 환경에 URL을 설정한다. 그 전까지 수집은 물리적으로 불가능.
- **우회 금지 규칙을 CLAUDE.md 불변식으로**: 로컬 폴백 저장·임시 큐·디버그용 강제 켜기 모두 금지 — 그게 곧 무동의 수집이다.
- 클라이언트는 서버 수신 여부를 몰라도 된다 — 게이트는 서버(또는 수집 파이프라인 입구) 한 곳에.

## 개통 체크리스트 (URL 설정 = "개통" 이벤트로 취급)

- [ ] 동의서(`CONSENT_FORM.md`) 배포·회수 완료, 미동의자 처리 방침 확정
- [ ] {{기관 승인 — 예: 교육청 공문·IRB}} 수령
- [ ] 수집 스키마 완성 (반쪽 스키마로 개통 금지 — "첫날부터 완전한 데이터")
- [ ] 미동의자 제외 로직 검증 (동의자 명단 기반 필터 또는 전원 동의 확인)
- [ ] 철회 절차 동작 확인 (철회 → 기존 데이터 삭제 범위는 `DATA_RETENTION_AND_DELETION_POLICY.md`)

## 동의 상태의 정본

- 동의 여부는 {{운영 DB의 동의 테이블 / 서면 대장}}이 정본. 클라이언트 저장(localStorage 등)은 참고일 뿐 — 기기 바뀌면 사라진다.
- 동의 항목이 여러 개면(수집 동의 / 연구 활용 동의 / 제3자 제공) 각각 독립 플래그.

## 미동의 사용자 경험

- 미동의 = 기능 차단이 아니라 **수집만 차단** (서비스 이용은 동일). 수집이 기능 조건이 되면 동의가 강요가 된다.
