# 브라우저 검증 규약 (2모드 — 자동 E2E와 AI 조작 검증은 서로 대체재가 아니다)

> §3 게이트(테스트·빌드)를 다 통과해도 **브라우저에서 안 돌아가는 변경은 흔하다** — 레이아웃 겹침, 라우팅 깨짐, 콘솔 에러, 모바일 뷰포트 잘림. 이 문서가 그 갭의 규약이다.

## 모드 ① 자동 E2E 스위트 (Playwright) — 회귀 방지, 커밋 게이트

- **역할**: 이미 확인된 핵심 여정이 **계속** 돌아감을 보장. `npx playwright test`가 §3 루프의 한 줄.
- 스모크는 치명 경로만(로드·핵심 여정·저장 유지 + 콘솔 에러 0) — 느려지면 게이트에서 빠지기 시작한다.
- 새 여정이 "핵심"으로 승격되면 스모크에 추가. E2E가 잡은 버그는 반드시 그 시나리오를 스위트에 박제.
- **한계**: 단언한 것만 본다. "떴지만 이상해 보임"·"버튼이 겹침"은 못 잡는다 → 모드 ②.

## 모드 ② AI 조작 검증 (playwright MCP / Chrome) — 신규 UI의 육안 확인

- **역할**: UI가 **새로 생기거나 바뀐** 커밋에서, AI가 직접 브라우저를 열어 사람 눈 역할을 한다. verify 스킬 4번(런타임 확인)의 UI 실행 방법.
- **절차** (AI 세션이 수행):
  1. 개발 서버 기동(또는 실행 중 재사용) → 변경된 화면으로 이동.
  2. **변경된 흐름을 실제 조작** — 클릭·입력·이동을 사용자처럼. 성공 경로 + 실패 경로(빈 입력·잘못된 값) 1개 이상.
  3. **스크린샷을 찍어 육안 판정** — 겹침·잘림·깨진 이미지·이상한 줄바꿈. 데스크톱 + 모바일 뷰포트(예: 375×667) 각 1장.
  4. **콘솔·네트워크 확인** — 에러 0, 실패 요청 0 (경고는 보고).
  5. 판정과 스크린샷 근거를 완료 보고에 포함. **"렌더됐다"가 아니라 "조작해서 끝까지 갔다"가 통과 기준.**
- 도구가 없으면: `claude mcp add playwright -- npx @playwright/mcp@latest` (INSTALL.md §6). MCP가 안 되는 환경이면 사용자에게 수동 확인을 요청하고 확인 항목 체크리스트를 제시하라 — 말없이 생략 금지.

## 어느 모드를 쓰나

| 상황 | 모드 |
|---|---|
| 모든 커밋 (§3 게이트) | ① 자동 E2E |
| UI 신규/변경 커밋 | ① + ② 둘 다 |
| 배포 직후 | ①을 배포 URL로 (`E2E_BASE_URL=…`) — deploy-verification-loop.md |
| 버그 재현 (debug 스킬 1번) | ② 로 재현 → 재현되면 ①에 시나리오 박제 |

## 합리화 방지

| 유혹 | 규율 |
|---|---|
| "컴포넌트 테스트 통과했으니 화면도 정상" | 레이아웃·라우팅·전역 CSS는 브라우저에서만 깨진다. UI 변경 = 모드 ② 필수. |
| "스크린샷 찍었고 떠 있으니 통과" | 조작해서 여정 끝까지. 정지 화면은 반쪽 검증. |
| "콘솔 경고쯤은…" | 에러 0이 기준. 경고는 보고 후 사용자 판단. |
| "E2E가 느려서 이번만 스킵" | 게이트 형해화의 시작. 느리면 스모크를 줄여라 — 스킵하지 말고. |
