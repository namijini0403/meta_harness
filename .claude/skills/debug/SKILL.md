# 체계적 디버깅 (debug)

**증상을 보자마자 코드를 고치고 싶은 충동이 디버깅 실패의 원인이다.** 순서를 강제한다.

## 절차 (건너뛰기 금지)

1. **재현 먼저**: 버그를 결정적으로 재현하는 최소 절차/입력을 확보한다. 재현 못 하면 수정도 검증도 불가 — 재현이 안 되면 로그·상태 수집부터.
   - 가능하면 재현을 **실패하는 테스트**로 박제한다 (수정 후 이 테스트가 통과 = 수정 증명 + 영구 회귀 방지).
2. **관찰**: 에러 메시지·로그·스택을 **끝까지 정독** (첫 줄만 보고 패턴매칭 금지). 실제 값 vs 기대 값을 명시적으로 적는다.
3. **가설 수립**: "○○ 때문에 △△가 된다"를 한 문장으로. 가설이 여러 개면 **가장 싸게 검증되는 것부터** (이진 탐색: 절반 지점에 로그/단언).
4. **가설 검증**: 코드를 고치지 말고 가설을 확인하는 관찰(로그·디버거·최소 실험)부터. 가설이 틀리면 2로 복귀 — 틀린 가설 위의 수정은 우연히 통과하는 시한폭탄.
5. **최소 수정**: 확인된 원인만 고친다. "겸사겸사 리팩터링" 금지 (수정 검증이 오염된다).
6. **회귀 확인**: 1의 재현 절차 통과 + CLAUDE.md §3 전체 게이트 (다른 걸 깨뜨리지 않았는가).
7. **기록**: 원인이 규약 위반·설계 결함이면 합리화 방지표 또는 CLAUDE.md §2/§8에 기록 — 같은 버그의 재발 방지가 수정보다 가치 있다.

## 시스템 상태를 바꾸는 명령 전 확인

재시작·삭제·설정 변경 전에: 증거가 **그 특정 조치**를 지지하는가? 알려진 장애와 패턴이 같아 보여도 원인이 다를 수 있다 — 인시던트 러닝북(있으면)의 증상→절 매핑부터.

## 합리화 방지

| 유혹 | 규율 |
|---|---|
| "딱 보니 ○○네, 바로 고치자" | 가설은 검증 후에만 수정 근거가 된다 (4번). |
| "재현이 번거로우니 고치고 나서 확인" | 재현 없는 수정은 검증 불가 — 순서 고정 (1번). |
| "고친 김에 주변도 정리" | 최소 수정만 (5번). 정리는 별도 커밋. |
| "테스트 하나 실패하지만 내 수정과 무관" | 원인 규명 전 커밋 금지 (6번). |
