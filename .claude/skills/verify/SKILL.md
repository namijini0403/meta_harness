# 커밋 전 검증 (verify)

CLAUDE.md §3의 검증 루프를 **순서대로, 전부** 실행한다. 하나라도 실패하면 커밋 금지 — 고치고 처음부터 재실행.

## 절차

1. **전체 게이트 실행** (CLAUDE.md §3의 명령을 순서대로):
   - 타입체크 → 전체 테스트 → 빌드 → 추가 게이트
   - ⚠️ 부분 실행(`-t` 필터)으로 통과한 것은 통과가 아니다. 전체 실행이 정본.
2. **기준선 대조**: 테스트 수를 **CLAUDE.md §3이 가리키는 정본**(§3에 직접 적혀 있으면 그 수치, "정본 = 상태판"이라 적혀 있으면 해당 마스터플랜 상태판)과 대조. **줄었으면** 뭔가 깨진 것 — 원인 규명 전 커밋 금지. 늘었으면 정본을 갱신.
3. **임시 파일 청소**: 검증용 임시 테스트·덤프 파일 전부 삭제 확인 (`git status`로 의도치 않은 파일 확인).
4. **런타임 확인** (제품 코드 변경 시): 테스트 통과 ≠ 동작. 변경된 흐름을 실제로 한 번 구동해 눈으로 확인.
   - **UI 변경이면 브라우저에서**: 변경 화면을 실제 조작(클릭·입력)으로 여정 끝까지 + 콘솔 에러 0 + 스크린샷 육안(데스크톱·모바일 뷰포트). e2e-deploy 팩 사용 시 상세 규약 = `docs/browser-verification.md`(모드 ②), 없으면 수동으로라도 — 말없이 생략 금지.
   - **배포를 동반하면**: 푸시 = 완료가 아니다 — `docs/deploy-verification-loop.md`(배포 대기 폴링 + 배포본 스모크) 통과 후에만 "배포 완료" 보고.
5. **커밋**: CLAUDE.md §7 규약(메시지 언어·따옴표 규칙·Co-Authored-By·명시 경로 add — `git add -A` 금지).

## 합리화 방지

| 유혹 | 규율 |
|---|---|
| "내 부분만 테스트 돌려서 통과했다" | 전체 실행만 정본. 부분 통과는 근거 아님. |
| "테스트가 통과했으니 동작한다" | 제품 코드 변경은 실제 구동 확인까지. |
| "급하니 빌드는 생략" | §3의 순서·전부가 게이트. 생략 = 미검증 커밋. |
| "테스트 수 줄었지만 내 작업과 무관할 것" | 원인 규명 전 커밋 금지. |
