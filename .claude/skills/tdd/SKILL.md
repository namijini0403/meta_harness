# 테스트 우선 개발 (tdd)

**구현 후에 쓴 테스트는 구현의 버그를 함께 통과시킨다.** 순서가 본질이다: 실패 확인 → 구현 → 통과.

## 절차 (RED → GREEN → REFACTOR)

1. **RED — 실패하는 테스트 먼저**
   - 스펙(계획서 §0 성공 기준)에서 검증 가능한 동작을 테스트로 옮긴다.
   - **실행해서 실패를 눈으로 확인** — 이 단계를 건너뛰면 "항상 통과하는 가짜 테스트"(단언 누락·잘못된 대상 호출)를 못 잡는다. 실패 메시지가 예상한 이유인지도 확인.
2. **GREEN — 통과하는 최소 구현**
   - 테스트를 통과시키는 데 필요한 만큼만. 미래 대비 일반화 금지 (YAGNI — 필요는 다음 테스트가 증명한다).
3. **REFACTOR — 통과 유지하며 정리**
   - 중복 제거·이름 개선. 매 정리 후 테스트 재실행.
4. 다음 동작으로 1~3 반복. 커밋 전 CLAUDE.md §3 전체 게이트.

## 테스트 설계 규칙

- **경계·오류 경로 우선**: 정상 경로 1개보다 경계값(0·빈 배열·최대치)·실패 경로가 버그를 잡는다.
- **동작을 검증, 구현을 검증하지 않기**: 내부 호출 순서·비공개 상태에 단언하면 리팩터링마다 깨지는 취성 테스트가 된다.
- **불변식은 속성 테스트로**: "모든 입력에서 ○○" 형태 규칙은 예시 몇 개가 아니라 대량 샘플 속성 테스트로 (verification-loop.md §3~4 커버리지 프로브·스모크 패턴 병용).
- 결정성: 시간·랜덤 의존은 주입(시드·클록)으로 통제 — 간헐 실패 테스트는 신뢰를 파괴한다.

## 적용 범위

- **적용**: 새 로직·버그 수정(debug 스킬 1번의 "재현 테스트 박제"가 곧 RED)·리팩터링 전 안전망.
- **완화 허용**: UI 미세 조정·일회성 스크립트·프로토타입(단 프로토타입이 살아남으면 테스트 소급).

## 합리화 방지

| 유혹 | 규율 |
|---|---|
| "간단한 코드라 테스트 없이" | 간단한 코드가 경계값에서 깨진다. 최소 1 경계 테스트. |
| "구현 다 됐으니 테스트는 형식상" | 사후 테스트는 RED 확인 불가 — 최소한 구현을 잠시 깨뜨려 테스트가 실패하는지 확인(뮤테이션 스모크). |
| "테스트가 빨간데 일단 커밋" | RED 상태 커밋 금지 (§3 게이트). |
