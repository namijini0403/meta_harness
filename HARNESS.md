# 하네스 엔지니어링 방법론

> 이 문서는 "왜 이렇게 설계했는가"를 설명한다. "무엇을 하라"는 CLAUDE.md와 각 팩에 있다.
> 새 프로젝트에 복사할 때 이 파일은 **선택** — 규약의 근거가 궁금할 때만 참조.

## 1. 하네스란 무엇인가

**하네스(harness) = AI가 프로젝트에서 작업할 때 따르는 강제 규약의 총체.**

구성 요소:
1. **프로토콜 문서** (CLAUDE.md) — 세션 시작 시 반드시 읽는 규약. 오케스트레이션 원칙, 검증 루프, 불변 결정, 커밋 규약.
2. **불변식 + 자동 강제** — 위반하면 테스트/CI가 실패하는 규칙. 문서 규칙의 최종 진화형.
3. **에이전트 팀 정의** (.claude/agents/) — 역할·소유 경계·핸드오프 프로토콜.
4. **스킬** (.claude/skills/) — 반복 작업의 파이프라인을 절차서로 고정. 넣었다 뺐다 가능한 단위.
5. **실패 기록** (합리화 방지표·결정 로그) — 하네스가 자라는 메커니즘.

핵심 명제: **"모델이 좋아서 품질이 나오는 게 아니라, 하네스가 좋아서 어떤 모델이든 품질이 나온다."**

## 2. 왜 오케스트레이터/서브에이전트 분리인가

- **컨텍스트 경제**: 대량 구현(수백 파일 콘텐츠 저작, 반복 배선)을 메인 컨텍스트에서 하면 계획·검증에 쓸 컨텍스트가 소진된다. 구현은 값싼 모델의 서브에이전트로 밀어내고, 메인은 계획→브리프 작성→산출물 검증→통합→커밋만 한다.
- **책임 비대칭**: 구현은 되돌리기 쉽다(파일 수정). 검증 통과 판정·설계 결정·커밋은 되돌리기 비싸다. 비싼 결정은 절대 위임하지 않는다.
- **모델 배정 원칙**: 설계·검수·승인 = 최상위 모델. 대량 저작·태깅·배선 = 중간 모델. 커밋·최종 판단 = 메인.

## 3. 병렬 안전의 3층 구조

병렬 에이전트의 충돌은 하네스 실패의 최대 원인이다. 3층으로 막는다:

1. **소유 경계** (agents/README.md의 표): 에이전트마다 "편집 가능한 파일 glob"을 명시. 경계 밖 파일이 필요하면 STOP + 보고 (직접 수정 금지).
2. **공유 순차 파일 목록**: 여러 에이전트가 append하는 파일(인덱스·레지스트리·라우팅 테이블)을 명시적으로 열거하고, 오케스트레이터가 순차 디스패치.
3. **브리프에 경계 내장**: 위임 브리프(8요소)에 파일 경계를 매번 다시 적는다. "에이전트가 README를 읽겠지"는 실패한다 — 브리프가 유일하게 확실한 전달 채널.

## 4. 불변식의 수명 주기

```
관찰된 버그 → 문서 규칙 (CLAUDE.md §불변식)
           → 체크리스트 항목 (착수 전 필독 문서)
           → 자동 테스트 (위반 = CI 실패)   ← 최종 형태
```

문서 규칙은 컨텍스트에서 밀려나면 잊힌다. 체크리스트는 "읽었다 치고" 넘어간다.
**테스트만이 잊혀도 안전하다.** 새 규칙이 생기면 "이걸 테스트로 만들 수 있나?"를 항상 물어라.

예 (DRACONIS 실전): "해설의 조사(받침) 오류 금지" → 처음엔 문서 규칙 → 42건 위반 발견 → josaAudit 테스트로 전환 → 이후 위반 0.

## 5. 완성 세트 원칙

작업 단위를 "커밋 가능한 완성 세트"로 정의한다. 예:
- 콘텐츠 프로젝트: 생성기 + 해설 + 오답 코딩 + 메타 태그 + 독립 검증 = 1세트
- API 프로젝트: 엔드포인트 + 검증 스키마 + 테스트 + 문서 = 1세트

세트의 일부만 만들고 커밋하면 "나중에 채우겠지"가 영원히 안 채워진다.
**세트 정의는 CLAUDE.md에, 강제는 가능하면 테스트로** (예: "메타 없는 항목 = 커버리지 프로브 실패").

## 6. 검증의 3중 구조

자동 테스트만으로는 부족하다. 층위가 다른 3가지를 조합한다:

| 층 | 무엇 | 잡는 것 |
|---|---|---|
| A. 속성 테스트 (자동) | 불변식을 대량 샘플로 강제 | 형식 위반·재현성·경계값 |
| B. 독립 재계산 (적대) | 산출물에서 **구현 코드를 안 보고** 정답을 다시 계산해 대조 | "표시값과 정답이 따로 노는" 부류 — 자동 테스트가 못 보는 실버그 |
| C. 육안 확인 (생략 금지) | 사용자가 실제로 보는 화면/텍스트를 덤프해 직접 읽기 | 문법·의미·톤·시각 정합 — 기계가 판정 못 하는 품질 |

B가 특히 중요하다: 생성 코드와 검증 코드를 같은 사람이(같은 로직으로) 짜면 같은 버그를 공유한다.
검증자는 **표시된 값에서 처음부터 다시 푼다.**

## 7. 세션 간 연속성

- **불변 결정 로그**: 사용자가 명시한 결정("보상은 드물게", "이 키는 변경 금지")을 날짜와 함께 기록. 다음 세션이 "개선"이랍시고 뒤집는 것을 막는다.
- **이어받기 프로토콜**: 에이전트가 중간에 죽으면(API 오류·한도) 남긴 파일을 확인하고 "기존 산출물 검수부터"를 지시하는 이어받기 에이전트를 재가동.
- **상태판**: 다단계 마스터플랜이 있으면 정본 문서에 Phase 상태판을 두고, 완료한 세션이 갱신할 의무를 규약에 명시.

## 8. 스킬의 설계 (넣었다 뺐다 가능하게)

스킬 = 반복 작업의 절차서 + (선택) 실행 가능한 검증기. 원칙:

- **자기완결**: 스킬 폴더 하나만 복사하면 동작. 프로젝트 특정 사실은 스킬 안에 하드코딩하지 말고 CLAUDE.md를 참조하게 한다.
- **트리거 명시**: frontmatter description에 "언제 이 스킬을 쓰는가"를 사용자 발화 예시로 적는다.
- **짝 스킬 링크**: 저작 스킬 ↔ 감사 스킬처럼 짝이 있으면 서로 명시 참조.
- **검증기 동봉**: 스킬이 산출물 형식을 정의하면, 그 형식을 검사하는 스캐너/테스트를 스킬 폴더에 동봉 (`scanners/`, `verifiers/`).

## 9. 하네스를 키우는 법 (유지보수 = 이것뿐)

1. 실패가 나면 → **합리화 방지표**에 한 줄 추가 (유혹 → 벌어진 일 → 규율).
2. 규율이 반복 적용되면 → CLAUDE.md 불변식으로 승격.
3. 불변식이 기계 판정 가능하면 → 테스트로 전환.
4. 사용자가 결정을 내리면 → 결정 로그에 날짜와 함께 기록.
5. 반복 작업이 3회 이상 나오면 → 스킬로 고정.

이 다섯 가지 외의 하네스 문서 작업(대규모 리라이트, 형식 통일)은 대부분 낭비다. 하네스는 실패 기록으로 자란다.
